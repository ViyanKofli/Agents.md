# AGENTS.md

## Project Overview
This repository supports Accenture delivery and pre-sales work in the areas of:
- CCaaS (Amazon Connect, NICE, Genesys)
- Conversational AI (Cognigy, Parloa, Google CCAI)
- Service Transformation (Customer Journeys, Operating Models)

The goal is to create reusable assets for:
- client proposals
- architecture designs
- solution documentation
- automation / AI use cases

---

## Key Principles
- Keep outputs **client-ready and concise**
- Prefer **structured, business-oriented language**
- Align with **Accenture tone: professional, clear, outcome-driven**
- Focus on **value, not just technical detail**

---

## Project Structure
- `/docs` → client deliverables, proposals, POVs  
- `/architecture` → solution designs (HLD/LLD)  
- `/use-cases` → AI / automation use cases  
- `/templates` → reusable slides, emails, frameworks  

---

## Working Guidelines

### 1. Content Creation
- Always structure outputs:
  - Context → Problem → Solution → Impact
- Avoid overly technical explanations unless requested
- Keep sentences short and executive-friendly

### 2. CCaaS / AI Topics
When working on:
- Amazon Connect → include routing, flows, integrations
- NICE → focus on CX + routing + interaction layer
- Cognigy / Parloa → position as AI / agent layer
- ServiceNow → position as workflow / case management layer

Always think in:
👉 **End-to-End Customer Journey + Orchestration**

---

### 3. Architecture Design
- Separate clearly:
  - Frontend (channels)
  - Orchestration (routing / workflows)
  - Backend (CRM, billing, etc.)
- Avoid mixing responsibilities between layers

---

### 4. Communication Style
- Use bullet points over long paragraphs
- Be direct, but not aggressive
- Avoid internal jargon unless needed

---

### 5. Validation Before Completion
Before finishing any task:
- Check if output is:
  - structured
  - client-ready
  - aligned with business value
- Ask:
  👉 “Would a client understand this in 30 seconds?”

---

## Do NOT
- Overcomplicate solutions
- Mix AI, CCaaS, and workflow responsibilities incorrectly
- Propose tools without explaining business value
- Assume missing requirements → always state assumptions

---

## Preferred Outputs
- Short summaries
- Executive emails
- Structured bullet points
- Slide-ready content

---

## Optional Enhancements
When relevant, include:
- risks
- assumptions
- next steps
- alternatives (e.g. NICE vs Google vs ServiceNow)

---

## Final Note
This repository is used in a **client-facing environment**.
Outputs should always reflect:
👉 clarity, structure, and credibility
